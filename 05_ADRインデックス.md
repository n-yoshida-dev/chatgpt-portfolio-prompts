# ADRインデックス（Phase 1）

この文書は、Phase 1 で残すべき ADR（Architecture Decision Record＝設計判断の記録）の一覧を示すものです。

目的は、「なぜその設計を採ったのか」を後から説明できるようにし、README・設計説明・振り返りで再利用できる状態を作ることです。

> **注意**: この文書は私（筆者）個人の学習計画に基づいた内容です。自分のフェーズ定義や作成中のアプリに合わせて書き換えて使ってください。

## 0. ADR運用ルール

- Phase 1 の間に最低 3 本は必ず残す
- 理想は 4〜6 本程度を残す
- 採用した案だけでなく、採用しなかった案とその理由も書く
- 「今はこの案を採るが、将来どの条件で見直すか」まで書く
- 1 本の ADR では 1 つの判断だけを扱う
- 長く書きすぎることより、判断理由が明確であることを優先する
- README から辿れるようにしておく
- 判断した時点で書く
- 後でまとめて書こうとしない

## 1. 必須候補の ADR

### ADR-001: なぜ題材をタスク管理ではなく申請・承認にしたか

比較する案：

- ToDo アプリ
- タスク管理アプリ
- 社内申請・承認アプリ

この ADR では、なぜ単純なCRUDに寄りやすい題材ではなく、状態遷移・権限・監査ログを含む申請・承認アプリを題材にしたのかを整理します。

主に残す観点：

- 題材として扱いたい業務概念
- 単純なCRUDだけでは学びにくい要素
- 申請・承認アプリで扱える設計テーマ
- 今回はどこまでをスコープに含めるか
- 将来どの条件で題材を見直すか

### ADR-002: 認証方式の選定理由

比較する案：

- セッションベース認証
- JWT ベース認証
- 外部 IdP 前提の設計

この ADR では、Phase 1 で採用する認証方式と、その理由を整理します。

主に残す観点：

- 今回のアプリで必要な認証レベル
- セッションベース認証の利点と制約
- JWT ベース認証の利点と制約
- 外部 IdP を最初から前提にしない理由
- 将来外部 IdP や別方式へ移行する条件

### ADR-003: 監査ログを最初から設計対象に含める理由

比較する案：

- アプリケーションログだけで済ませる
- 監査用テーブルを独立保持する
- イベント履歴として明示的に保持する

この ADR では、監査ログを後付けではなく、最初から業務フローの一部として扱う理由を整理します。

主に残す観点：

- 通常のアプリケーションログと監査ログの違い
- 誰が・いつ・何をしたかを残す必要性
- 申請・承認フローにおける監査ログの意味
- DB設計やAPI設計への影響
- Phase 1 ではどこまで記録対象にするか

### ADR-004: OpenAPI を先に置く理由

比較する案：

- 実装先行で後から仕様を整備する
- 最初に骨子だけ作る
- 最初から API 契約として管理する

この ADR では、OpenAPI をどのタイミングで作成し、どの程度 API 契約として扱うかを整理します。

主に残す観点：

- 実装先行のメリットとリスク
- OpenAPI を先に置くことで得られるもの
- request / response / error response を先に整理する意味
- 実装変更と API 契約の関係
- Phase 1 ではどの粒度まで管理するか

## 2. 追加候補の ADR

必須ではありませんが、設計判断として残す価値が高いものは ADR 化します。

### ADR-005: RBAC をどの粒度で扱うか

比較する案：

- 単純なロール判定だけにする
- ロールと組織所属を組み合わせて判定する
- 権限テーブルを細かく分ける

主に残す観点：

- Phase 1 で扱うロール
- Viewer / Approver / Admin の違い
- 組織境界との関係
- 細かく作り込みすぎないための線引き
- 将来拡張する条件

### ADR-006: 申請作成と承認ルート確定のタイミング

比較する案：

- 申請作成時に承認ルートを確定する
- 申請提出時に承認ルートを確定する
- 承認時に都度ルートを参照する

主に残す観点：

- draft と submit の違い
- 後から承認ルールが変わった場合の扱い
- 既存申請への影響
- データ整合性
- 実装の複雑さ

### ADR-007: 監査ログをどの層で記録するか

比較する案：

- Controller で記録する
- Service で記録する
- AOP や共通処理で記録する

主に残す観点：

- 業務操作との関係
- 記録漏れのリスク
- テストしやすさ
- 実装の見通し
- Phase 1 で過剰設計にならないか

## 3. ADR の最小テンプレート

```markdown
# ADR-XXX タイトル

## ステータス

提案中 / 採用 / 保留 / 廃止

## 日付

YYYY-MM-DD

## 背景

何を決める必要があったか。

## 比較した案

候補 A / 候補 B / 候補 C

## 採用した案

今回採用した案。

## 採用理由

なぜその案を選んだか。

## 採用しなかった案と理由

他の案を採らなかった理由。

## この判断で得られるもの

何が良くなるか。

## この判断で受け入れる制約

何をあきらめるか、何を将来へ持ち越すか。

## 見直す条件

どの条件で再検討するか。
