# フェーズ1専用学習対象定義

この文書は、Phase 1 で「何を学ぶか」「何を作るか」「まだ優先しないものは何か」を明確にする。
Phase 1 の完了に必要なバックエンドの基盤を先に固め、学習テーマが広がりすぎることを防ぐ。

> **注意**: この文書は私（筆者）個人の学習計画に基づいた内容です。そのまま使うのではなく、自分のロードマップに合わせて書き換えて使ってください。

---

## 0. この文書の位置づけ

Phase 1 は学習ロードマップ全体の土台を作るフェーズである。
ここで作るべきものは機能数が多いアプリではなく、React・AWS・Terraform・Go への置き換えに進む前提になる、説明可能なバックエンドの基盤である。

---

## 1. Phase 1 のゴール

Java / Spring Boot で OrgFlow API v1 を作り、「B2B ワークフロー SaaS のバックエンドを、設計理由まで含めて説明できる状態」にすること。

完了条件:

- 組織、ユーザー、ロール、申請、承認、監査ログが、それぞれ何を表す概念か説明できる
- API がログイン済みの利用者だけを受け付ける状態になっている
- ロールごとに実行できる操作が変わる
- 申請作成 → 承認または差し戻し → 監査ログ記録の流れが壊れずに動く
- OpenAPI が API 契約として管理されている
- 単体テストと統合テストがあり、CI で自動実行に成功する
- README と ADR を読めば、なぜその設計にしたかを説明できる

---

## 2. Phase 1 で扱う題材

B2B ワークフロー SaaS 風の「OrgFlow」。申請・承認・監査・権限・組織切替・データ分離を含む業務 SaaS。

### 最低限扱う業務概念

- `organizations`: 利用者や申請データが属する組織単位
- `users`: システムを利用する人
- `roles`: 利用者が実行できる操作の権限区分
- `requests`: 利用者が作成する申請データ
- `approvals`: 申請に対する承認・差し戻し・却下などの判断記録
- `audit_logs`: 誰が、どの組織で、いつ、どの操作を行ったかの証跡

### 最低限扱う業務フロー

ログイン → 組織切替 → 申請作成 → 承認 → 差し戻し → 監査ログ記録

---

## 3. Phase 1 で学習対象に含めるもの

### 3-1. ドメイン理解

コードを書く前に、利用者・ロール・申請・承認・差し戻し・監査ログ・組織切替の意味を整理する。

### 3-2. DB設計

業務概念の意味が崩れないこと、後から大きく壊れにくいことを優先する。

### 3-3. API設計

業務上の操作として説明できること、後から実装言語を変えても契約が保てることを優先する。

### 3-4. Java / Spring Boot 実装

Controller / Service / Repository の役割分担、DTO と Entity の境界、認証・認可の判定位置、監査ログ記録、更新競合への備えを扱う。

### 3-5. 認証・認可

ログイン済み API の成立、認証と認可の分離、ロール別の操作制御、403 の適切な返却を扱う。

### 3-6. RBAC

Viewer / Approver / Admin などのロール差分、ロール別の操作制御、API側でのアクセス制御を扱う。

### 3-7. 監査ログ

誰が・どの組織で・いつ・何をしたか・状態変更の内容を記録する。後付けの補助機能ではなく、最初から業務フローの一部として設計・実装する。

### 3-8. テスト

単体テスト、統合テスト、主要業務フロー検証、権限差分検証、監査ログ検証、409 Conflict 検証を扱う。

### 3-9. CI

build、test、CI の成功状態の維持を扱う。

### 3-10. OpenAPI

API 一覧、request / response、主要なエラー、将来の React / Go 進行時にも維持したい契約として扱う。

### 3-11. README / ADR

README: 起動方法、テスト方法、主要機能、技術選定の概要。
ADR: 題材選定理由、認証方式選定理由、監査ログを最初から含める理由、OpenAPI を先に置く理由。

---

## 4. Phase 1 でまだ主対象にしないもの

- React / TypeScript の実務品質 UI
- Go 実装（Phase 4）
- AWS 本番相当構成（Phase 3）
- Terraform の本格運用
- Kubernetes
- AI 機能をアプリの中心にすること

---

## 5. Phase 1 の学習順序

1. 題材理解とユースケース整理
2. ドメイン整理
3. DB設計
4. API の責務整理
5. OpenAPI の骨子作成
6. Spring Boot で API 実装
7. 認証・認可 / RBAC の組み込み
8. 監査ログ実装
9. 単体テスト / 統合テスト
10. Docker Compose での再現
11. GitHub Actions CI
12. README / ADR 更新

---

## 6. Phase 1 完了判定

- `docker compose up` で `api` / `db` が起動する
- ログイン、申請作成、承認、監査ログ記録が API で通る
- RBAC により、ロールごとに実行できる操作が変わる
- 単体テスト / 統合テストが CI で成功する
- OpenAPI と README が更新されている
- ADR が最低 3 本ある
