# フェーズ1専用学習対象定義

この文書は、Phase 1 で「何を学ぶか」「何を作るか」「まだ優先しないものは何か」を明確にするためのものです。

Phase 1 の完了に必要なバックエンドの基盤を先に固め、学習テーマが広がりすぎることを防ぎます。

> **注意**: この文書は私（筆者）個人の学習計画に基づいた内容です。そのまま使うのではなく、自分の学習テーマや作成中のアプリに合わせて書き換えて使ってください。

## 0. この文書の位置づけ

Phase 1 は、学習ロードマップ全体の土台を作るフェーズです。

ここで作るべきものは、機能数が多いアプリではありません。

まずは、業務系Webアプリのバックエンドとして必要になる、説明可能な基盤を作ることを優先します。

具体的には、次のような観点を扱います。

- 業務ドメインの整理
- RDB設計
- API設計
- Java / Spring Boot による実装
- 認証・認可
- ロール別の操作制御
- 監査ログ
- テスト
- CI
- README / ADR による説明

後続フェーズでフロントエンド、クラウド、IaC、別言語実装などに進むとしても、まずはバックエンドの設計・実装・説明の土台を固めることを目的とします。

## 1. Phase 1 のゴール

Java / Spring Boot で OrgFlow API v1 を作り、**業務系Webアプリのバックエンドを、設計理由まで含めて説明できる状態**にすることをゴールとします。

完了条件は以下です。

- 組織、ユーザー、ロール、申請、承認、監査ログが、それぞれ何を表す概念か説明できる
- API がログイン済みの利用者だけを受け付ける状態になっている
- ロールごとに実行できる操作が変わる
- 申請作成 → 承認または差し戻し → 監査ログ記録の流れが壊れずに動く
- OpenAPI が API 契約として管理されている
- 単体テストと統合テストがあり、CI で自動実行に成功する
- README と ADR を読めば、なぜその設計にしたかを説明できる

## 2. Phase 1 で扱う題材

Phase 1 では、組織内の申請・承認を扱うワークフロー管理アプリ「OrgFlow」を題材にします。

扱う主な要素は、申請、承認、監査、権限、組織切替、データ分離です。

### 最低限扱う業務概念

- `organizations`: 利用者や申請データが属する組織単位
- `users`: システムを利用する人
- `roles`: 利用者が実行できる操作の権限区分
- `requests`: 利用者が作成する申請データ
- `approvals`: 申請に対する承認・差し戻し・却下などの判断記録
- `audit_logs`: 誰が、どの組織で、いつ、どの操作を行ったかの証跡

### 最低限扱う業務フロー

ログイン → 組織切替 → 申請作成 → 承認 → 差し戻し → 監査ログ記録

## 3. Phase 1 で学習対象に含めるもの

### 3-1. ドメイン理解

コードを書く前に、利用者・ロール・申請・承認・差し戻し・監査ログ・組織切替の意味を整理します。

特に、単なるCRUDとして扱うのではなく、「誰が」「どの状態の申請に対して」「どの操作を行えるのか」を説明できるようにします。

### 3-2. DB設計

業務概念の意味が崩れないこと、後から大きく壊れにくいことを優先します。

主に以下を扱います。

- テーブル設計
- 主キー / 外部キー
- 一意制約
- NOT NULL 制約
- ステータス管理
- 参照関係
- 履歴・監査情報
- 論理削除の要否
- 将来の検索条件を見据えた index 候補

### 3-3. API設計

業務上の操作として説明できること、後から実装の内部構造が変わっても API 契約が保てることを優先します。

主に以下を扱います。

- API の責務
- パス設計
- HTTP メソッド
- request / response
- エラー設計
- 認証・認可との関係
- OpenAPI での表現

### 3-4. Java / Spring Boot 実装

Java / Spring Boot を使って、業務系Webアプリのバックエンド実装に必要な基本要素を扱います。

主に以下を学習対象に含めます。

- Controller / Service / Repository の役割分担
- DTO と Entity の境界
- 入力値検証
- 例外処理
- 認証・認可の判定位置
- 監査ログ記録
- 更新競合への備え
- トランザクション境界

### 3-5. 認証・認可

ログイン済み API の成立、認証と認可の分離、ロール別の操作制御、403 の適切な返却を扱います。

ここでは、単に「ログインしているか」だけでなく、**その利用者が対象データに対して操作できるか**まで考えます。

### 3-6. RBAC

Viewer / Approver / Admin などのロール差分、ロール別の操作制御、API側でのアクセス制御を扱います。

主に以下を整理します。

- ロールごとにできる操作
- 認証済みであれば許可される操作
- 特定ロールだけが実行できる操作
- 所属組織や対象データによって判定が変わる操作

### 3-7. 監査ログ

誰が・どの組織で・いつ・何をしたか・状態変更の内容を記録します。

監査ログは後付けの補助機能ではなく、業務フローの一部として設計・実装します。

主に以下を扱います。

- 記録対象の操作
- 操作者
- 対象リソース
- 操作種別
- 操作日時
- 状態変更前後の情報
- 監査ログをどの層で記録するか

### 3-8. テスト

単体テスト、統合テスト、主要業務フロー検証、権限差分検証、監査ログ検証、409 Conflict 検証を扱います。

主に以下を確認します。

- 正常系
- 異常系
- 境界値
- 権限差分
- 状態遷移
- 監査ログの記録
- 更新競合
- API と DB を含めた統合的な動作

### 3-9. CI

build、test、CI の成功状態の維持を扱います。

主に以下を対象にします。

- GitHub Actions による自動実行
- build の成功確認
- test の成功確認
- 失敗時に原因を追いやすい構成
- 過剰に複雑にしすぎないワークフロー

### 3-10. OpenAPI

OpenAPI は、API 一覧、request / response、主要なエラー、今後も維持したい API 契約として扱います。

主に以下を管理します。

- エンドポイント一覧
- HTTP メソッド
- request body
- response body
- status code
- error response
- 認証が必要なAPIかどうか

### 3-11. README / ADR

README では、起動方法、テスト方法、主要機能、技術選定の概要を整理します。

ADR では、設計判断の理由を記録します。

Phase 1 では、少なくとも以下のような論点を ADR として残すことを想定します。

- 題材選定理由
- 認証方式選定理由
- 監査ログを最初から含める理由
- OpenAPI を先に置く理由
- RBAC をどの粒度で扱うか

## 4. Phase 1 でまだ主対象にしないもの

Phase 1 では、バックエンドの基盤作りに集中します。

そのため、以下はまだ主対象にしません。

- React / TypeScript の高度な UI 実装
- 別言語による再実装
- クラウド本番相当構成
- Terraform の本格運用
- Kubernetes
- AI 機能をアプリの中心にすること

これらは重要なテーマですが、Phase 1 で同時に扱うと学習範囲が広がりすぎます。

まずは、Java / Spring Boot によるバックエンドの設計・実装・説明を優先します。

## 5. Phase 1 の学習順序

Phase 1 は、次の順番で進めます。

1. 題材理解とユースケース整理
2. ドメイン整理
3. DB設計
4. API の責務整理
5. OpenAPI の骨子作成
6. Spring Boot で API 実装
7. 認証・認可 / RBAC の組み込み
8. 監査ログ実装
9. 単体テスト / 統合テスト
10. Docker Compose での再現
11. GitHub Actions CI
12. README / ADR 更新

この順番にする理由は、実装から先に入ると、後からDB設計やAPI責務の見直しが大きくなりやすいからです。

先に業務概念、DB、API契約を整理し、そのうえで実装・テスト・CI・説明資料に進みます。

## 6. Phase 1 完了判定

Phase 1 は、次の状態になったら完了とします。

- `docker compose up` で `api` / `db` が起動する
- ログイン、申請作成、承認、監査ログ記録が API で通る
- RBAC により、ロールごとに実行できる操作が変わる
- 単体テスト / 統合テストが CI で成功する
- OpenAPI と README が更新されている
- ADR が最低 3 本ある
- 主要な設計判断について、自分の言葉で説明できる
